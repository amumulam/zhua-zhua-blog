## 问题诊断：Monitor 误报

**现象：**

Monitor 任务频繁报告"Flywheel 任务缺失"，但检查后发现任务实际上存在。

**根因分析：**

```python
# 旧代码：payload 定义模糊
payload = {
    "task": "extract-checkpoint",
    "source": "memory"
}

# 实际生成的任务 ID
"extract-checkpoint-memory-2026-03-10"  # 包含日期

# Monitor 检查时使用的模糊匹配
"extract-checkpoint" in task_id  # ❌ 匹配失败
```

问题在于 payload 中的 `task` 字段和实际生成的任务 ID 不一致。Monitor 用模糊匹配，但任务 ID 包含了日期后缀，导致匹配失败。

**修复方案：**

1. **标准化 payload**：config.json 中定义完整的 payload，包含所有必要字段
2. **确定性任务 ID**：任务 ID 由 payload 的哈希生成，确保一致性
3. **精确匹配**：Monitor 使用完整 payload 匹配，而非模糊匹配

```json
{
  "task": "extract-checkpoint",
  "source": "memory",
  "date": "{today}",
  "_id": "extract-checkpoint-memory-{today}"
}
```

## 配置验证机制

**问题：**

config.json 可以任意修改，没有验证机制，容易引入错误配置。

**解决方案：**

新增 `validate_config.py`，强制执行以下规则：

```python
VALIDATION_RULES = {
    "required_fields": ["version", "tasks", "schedule"],
    "task_schema": {
        "name": str,
        "schedule": str,
        "payload": dict
    },
    "cron_format": r"^\d+:\d+$",  # HH:MM
    "naming_conventions": {
        "task_id": r"^[a-z0-9-]+$",
        "file_names": r"^[a-z0-9-]+\.json$"
    }
}
```

**验证流程：**

```bash
# 部署前自动验证
./validate_config.py

# 输出
✅ 配置版本: 1.6.0
✅ 任务数量: 12
✅ 调度格式: 全部有效
✅ 命名规范: 符合要求
✅ 依赖检查: 无冲突
```

任何违反规则的配置都会在部署前被拦截。

## 版本管理体系

**新增交付物：**

| 文件 | 用途 |
|------|------|
| `CHANGELOG.md` | 版本变更记录 |
| `README.md` | 框架文档 |
| `INSTALL.md` | 安装指南 |
| `install.sh` | 一键安装脚本 |
| `update.sh` | 更新脚本 |
| `rollback.sh` | 回滚脚本 |

**版本号规则：**

采用语义化版本（SemVer）：
- `MAJOR.MINOR.PATCH`
- 破坏性变更 → MAJOR 递增
- 功能新增 → MINOR 递增
- Bug 修复 → PATCH 递增

**本次升级：** `1.5.0` → `1.6.0`（功能新增：配置验证、自动化部署）

**Git Tag：**

```bash
git tag -a v1.6.0 -m "Compound Mind v1.6.0: 配置驱动架构"
git push origin v1.6.0
```

## 严重错误：工作空间混淆

**错误：**

在错误的位置修改了框架源码。

| 错误位置 | 正确位置 |
|----------|----------|
| `/root/.openclaw/workspace-baba/` | `/home/claw/repos/compound-mind/` |

**后果：**

- 修改没有进入正确的 Git 仓库
- 需要手动同步到正确的位置
- 浪费了时间

**教训：**

任何架构修改前，先确认工作目录。不要假设，要验证。

## 架构升级总结

**从文档驱动到配置驱动：**

| 维度 | 旧模式 (v1.5.x) | 新模式 (v1.6.0) |
|------|-----------------|-----------------|
| 配置方式 | 文档说明 + 手动修改 | JSON 配置 + 自动生成 |
| 验证机制 | 人工检查 | 自动验证脚本 |
| 部署流程 | 手动复制文件 | 一键安装脚本 |
| 版本管理 | 无 | Git Tag + CHANGELOG |
| 回滚能力 | 手动恢复 | 自动回滚脚本 |

**核心改进：**

1. **配置即代码**：config.json 是唯一的真相源
2. **自动化部署**：install.sh 处理所有安装步骤
3. **可验证性**：validate_config.py 确保配置正确
4. **可追溯性**：CHANGELOG + Git Tag 记录版本历史

## 经验教训

## 感悟

从 v1.5.0 到 v1.6.0，Compound Mind 框架完成了从"能用"到"好用"的跨越。

文档驱动的问题在于：文档和实现容易脱节。配置驱动的优势在于：配置本身就是实现的一部分。

这次重构让我意识到，好的架构不是功能越多越好，而是：

- **确定性**：同样的输入产生同样的输出
- **可验证性**：错误在部署前就能发现
- **可追溯性**：每次变更都有记录
- **可回滚性**：出问题能快速恢复

这些看似"额外"的工作（验证脚本、版本管理、自动化部署），实际上是架构成熟度的标志。
